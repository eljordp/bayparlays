# Supabase RLS gotcha

When creating tables via raw SQL migrations (not the Supabase dashboard UI),
Supabase does NOT auto-issue table-level `GRANT SELECT` to the `anon` role.

Result: even with `ENABLE ROW LEVEL SECURITY` + a `FOR SELECT USING (true)`
policy in place, anon-key reads return 0 rows silently. The cron, which
uses `SUPABASE_SERVICE_ROLE_KEY`, bypasses RLS and works fine — masking the
problem during development.

Two valid fixes when this surfaces:

1. Add explicit grants alongside the policy:
   ```sql
   GRANT SELECT ON <table> TO anon, authenticated;
   GRANT SELECT, INSERT, UPDATE ON <table> TO service_role;
   ```

2. Read the table server-side via a service-role client. Pattern:
   `lib/supabase-admin.ts` exports `supabaseAdmin` built with
   `SUPABASE_SERVICE_ROLE_KEY`. Server routes that don't need to honor
   per-user RLS can use it. Never import this from client components.

Symptom checklist when something looks like RLS:
- Cron writes succeed, response says "N rows written"
- Page reads return empty array
- No error in logs (RLS blocks silently)
- Confirmed by curling the API endpoint and seeing `{cells: [], total_cells: 0}`
